INFORMACIÓN SOBRE LO QUE MUESTRA LA LINEA ANTERIOR:
En la primera columna aparecen los diferentes rangos de memoria separados por un guion. La primera dirección del rango es la 
dirección virtual de comienzo, mientras que la segunda es la dirección final (más uno). todas estas direcciones se encuentran 
en la zona de memoria reservada para la tarea. el tamaño de cada una de las áreas mostradas es múltiplo de 4 KiB, el tamaño 
de la página. 
En la segunda columna aparecen los permisos de lectura (r), escritura (w) y ejecución (x) asociados al área de memoria. 
También se indica si el área de memoria es privada (p) o compartida por varias tareas (s). 
en la columna más a la derecha aparece el nombre del archivo asociado a cada área de memoria si hay alguno asociado, 
o bien un espacio en blanco para áreas anónimas gestionadas por el sistema

MAS INFO EN EL DOCUMENTO

CÓDIGO ESTÁTICO: código de la tarea presente en su archivo ejecutable con atributos r-xp
DATOS ESTÁTICOS: las constantes y variables globales de la tarea, también definidas en el archivo ejecutable con 
atributos r--p para constantes y rw-p para variables.
PILA(STACK): Almacena las variables locales de la tarea así como las direcciones de retorno de las funciones, la 
ubicación es aleatoria y la fija el S.O durante la carga de la tarea. Se corresponde con el rango de memoria 
marcado como [stack]
MONTÓN O MONTÍCULO(heap): Reservada para memoria solicitada dinámicamente. Se emplea la función malloc().
Se corresponde con el rango de memoria marcado como [heap]
BIBLIOTECAS DE ENLACE DINÁMICO: El ejecutable de la tarea contiene código que hace de interfaz con las bibliotecas 
dinámicas. El enlace con estas bibliotecas se hace en tiempo de carga o en tiempo de ejecución
Las bibliotecas dinámicas en Linux tienen típicamente la extensión .so
VDSO (Virtual Dynamic Shared Object): Se trata de una pequeña zona de memoria que contiene una librería dinámica, 
la cual permite implementar de forma muy eficiente llamadas a servicios que no requieren privilegios para ser 
ejecutados.
VVAR (Virtual Dynamic Shared Object): Zona de memoria de sólo lectura que contiene datos a los que acceden 
las rutinas almacenadas en el área VDSO.
LIBRERIA ESTÁTICA: Si se desea enlazar con la versión estática de la librería (libc.a) debe indicarse de forma 
explícita añadiendo la opción -static

¿Coinciden las direcciones virtuales de la variable local en dos instancias distintas del programa?
- No coinciden porque las variables locales se crean en la pila y, para evitar problemas de seguridad, 
el SO operativo evita colocar la pila siempre en las mismas direcciones virtuales.

¿Coinciden las direcciones virtuales de las variables globales de ambos programas?

variables del otro.

únicamente una copia en memoria del código ejecutable.

3. ASIGNACIÓN DINÁMICA DE MEMORIA
Se proporcionan servicios que permiten que una tarea, a lo largo de su ejecución, pueda pedir más memoria 
cuando la necesita y liberarla más adelante cuando ya no le hace falta.
Si la memoria se solicita al sistema operativo, éste puede establecer restricciones sobre la máxima 
cantidad de memoria que puede asignar a una tarea, así como la proporción de reparto de ésta entre 
la memoria física y el archivo de paginación. La memoria solicitada por una tarea al sistema operativo 
se asocia a un rango de direcciones virtuales que puede ser especificado por la tarea en el momento de 
llamar al servicio, aunque lo habitual es dejar libertad al sistema operativo para que elija el rango 
de direcciones virtuales de la tarea más apropiado.

3.1 PRIMERA FORMA DE LLEVAR A CABO LA PETICIÓN DE MEMORIA: EL SERVICIO MMPAP DE POSIX
POSIX engloba un conjunto de estándares de interfaz con el sistema operativo. Buscan que una aplicación 
pueda compilarse y ejecutarse directamente sobre diferentes sistemas operativos.

La función shm_open() crea un descriptor de memoria con un cierto nombre y unos parámetros de acceso al mismo. 
El nombre debe comenzar con el carácter "/"

La función ftruncate() configura el descriptor de memoria con un tamaño de 16 KiB. Una vez ejecutada esta 
función tenemos un descriptor de memoria configurado, pero la tarea aún no ha recibido los 16 KiB de memoria.

Parámetros empleados en la llamada a la función mmap():
if( p = mmap(NULL, 16*1024, PROT_READ|PROT_WRITE, MAP_SHARED, memd, 0)) == MAP_FAILED)
- NULL: El S.O elige la dirección virtual de comienzo. Si especificásemos una dirección, el S.O debería mapear
la memoria a partir de esa dirección virtual.
- 16*1024: Tamaño del área de memoria a mapear. No tiene por qué coincidir con el valor empleado en la llamada 
a ftruncate(), pues podríamos por ejemplo pedir primero 4 KiB y después pedir los 12 KiB restantes.
- PROT_READ|PROT_WRITE: Indica que el área de memoria que le asignará el S.O será de lectura y escritura.
- MAP_SHARED. Indica que la memoria puede ser accedida desde otras tareas. Permite la compartición del área 
de memoria solicitada por aquellas tareas que crearán sus descriptores de memoria usando el mismo nombre y 
eliminando la opción O_CREAT. Si se desea impedir la posibilidad de compartición con otra u otras tareas 
debería especificarse la opción MAP_PRIVATE.
- memd. Es el descriptor de memoria, previamente creado y configurado.
- 0. Especifica el desplazamiento dentro del elemento a mapear. En nuestro caso siempre será cero.
- Éxito-->la función devuelve la dirección virtual a partir de la cual el S.O ha mapeado la memoria solicitada.
En caso contrario, la función devuelve el valor MAP_FAILED.

3.3. Las funcionas malloc() y free()
Menos potentes que las funciones POSIX mmap() y mummap(), permiten llevar a cabo mucho más fácilmente la 
solicitud y liberación dinámica de memoria
La implementación de la función malloc() emplea el montón (heap) para ubicar la memoria solicitada
La función malloc() recibe como parámetro el número de bytes del área de memoria a solicitar y devuelve la 
dirección virtual de comienzo a partir de la cual se ubica dicho área de memoria. Cuando ocurre un error 
durante su ejecución devuelve NULL.
Una vez que el área de memoria solicitada ya no es necesaria se debe liberar, para lo cual se emplea la 
función free()

¿En qué zona de memoria de la tarea se encuentra ubicado el área de memoria solicitada?

decir, montón. Apreciando así, que se la ubicación se selecciona automáticamente

¿En qué zona se han asignado los 256 KiB de memoria solicitada?

decir, está en una zona sin nombre donde se debe reservar un área nueva.
Se asigna en una zona nueva sin nombre porque se ha solicitado una cantidad de memoria superior a la

4. LA TABLA DE PÁGINAS EN GNU/LINUX
La máquina empleada en prácticas emplea un esquema de paginación con dos tamaños de página posibles: 
4 KiB, que requiere traducción en dos pasos, y 4 MiB, que se puede traducir en un paso.

- Los bits 31 a 12 (Page Base Address) de la ETP contienen el marco de página y por lo tanto la traducción 
de la página virtual cuando el bit de presencia P está a 1. El resto de bits constituyen los flags, 
empleados para establecer las características de la página. Los flags más destacados son estos:
- Bit 0 (P). Se trata del bit de presencia. Cuando está a 1 la página está en memoria y los bits 31 a 12 
contienen un marco de página válido.
- Bit 1 (R/W). Cuando toma el valor 0 la página es de sólo lectura, mientras que cuando toma el valor 1 es 
de lectura y escritura.
- Bit 2 (U/S). Cuando toma el valor 0 la página es de supervisor, esto es, sólo puede acceder a la misma el 
sistema operativo. Cuando toma el valor 1 la tarea también puede acceder a ella.
- Bit 3 (PWT). Cuando está a 1 y está habilitada la caché indica que la página emplea una estrategia de 
escritura write-through para la página. Cuando está a cero emplea la estrategia write-back.
- Bit 4 (PCD). Cuando está a 1 la caché está deshabilitada para la página y por lo tanto cualquier acceso 
va directamente a memoria principal. En caso contrario, la caché está habilitada.
- Bit 5 (A). Se pone a 1 cuando la página es accedida, ya sea en lectura o en escritura. Una vez se pone 
a 1 no se pone a 0 hasta que el sistema operativo reinicia este bit escribiendo sobre la ETP.
- Bit 6 (D). Es análogo al bit de dirty de las cachés. Cuando se escribe en la página este bit se pone a 
1, de tal forma que si en un futuro es necesario reemplazar la página en memoria para llevar una nueva 
página, la página reemplazada no se descarta, sino que se guarda en disco.
- Bit 8 (G). Este bit cuando está a 1 marca la ETP como global. Por lo tanto, la ETP es válida para todas 
las tareas y no es necesario vaciar la copia de esta ETP en el TLB cada vez que se produce un cambio de 
contexto de una tarea a otra. Las páginas del sistema operativo suelen tener este bit a 1.

¿Por qué coinciden los flags de la pagina virtual de la variable global y local?
- Como ambas son páginas para albergar variables, ambas comparten las mismas propiedades, aunque se asignen 
en áreas de la memoria distintas, es decir, ambas son páginas de datos

¿Por qué no coinciden los flags de la página de memoria asignada a la función con las páginas asignadas 
a las variables de memoria?
- La página de memoria asignada a la función está prevista para alojar código. En la zona de código no 
se puede escribir por lo que el bit correspondiente no tiene que estar activo. En las páginas de las 
variables se alojaran datos, es decir, la asignada a la función es para código, y la de memoria para datos

¿Qué flags son diferentes respecto a las páginas de las variables? ¿Por qué? 
Se diferencia en dos bits:
- El de Lectura/Escritura ya que la página, la albergar código, es de solo lectura.
- El de escrita (dirty). Este bit indica que se ha escrito en esa página de memoria y, como es de solo 
  lectura, no es posible que está activo.
La respuesta correcta es: El bit de lectura/escritura y el de escrita (dirty)
